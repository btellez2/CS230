CS 230 Software Design Document

Project Overview

This repository contains my CS 230 portfolio artifact for The Gaming Room. The project includes a completed software design document for Draw It or Lose It, a game application that The Gaming Room wanted to expand from Android to multiple platforms. The purpose of this project was to analyze the client’s needs and create a clear software design plan that could guide future development.

Client and Software Requirements

The client was The Gaming Room. They wanted to design and expand their game, Draw It or Lose It, so it could work across multiple operating platforms. The game needed to support multiple teams, multiple players, unique game and team names, and a reliable structure for managing game sessions. The client also needed recommendations for operating platforms, memory management, storage management, security, and software architecture.

What I Did Well

I think I did well organizing the software design document and explaining the client’s requirements clearly. I was able to break down the main needs of the system and connect those needs to technical recommendations. I also explained why certain design choices, such as a client-server architecture, would help support the game across multiple platforms.

Helpful Parts of the Design Process

Working through the design document was helpful because it allowed me to understand the system before focusing on the code. The document helped me think about the structure of the application, the user requirements, and the technical decisions that needed to be made. Planning the design first made the development process easier because the main goals and requirements were already organized.

What I Would Revise

If I could revise one part of my work, I would improve the architecture section by adding more detail and making the explanation stronger. I would explain more clearly how the client-server model supports multiple users, multiple platforms, and future growth. I would also make sure the wording is easy to understand for both technical and nontechnical readers.

Interpreting User Needs

I interpreted the user’s needs by focusing on what The Gaming Room wanted the software to accomplish. The design needed to support gameplay, teams, players, unique names, and platform expansion. Considering the user’s needs is important because software should solve the client’s actual problem. If the design does not match the user’s needs, the final product may not be useful or successful.

Software Design Approach

My approach to designing the software was to first review the client’s requirements, then identify the main system features and technical needs. I used the design document to organize my ideas and explain the reasoning behind my recommendations. In the future, I would continue using requirements analysis, UML diagrams, architecture planning, and clear documentation when designing similar software applications.
